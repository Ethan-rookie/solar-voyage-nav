# OrbitGo 设计分类

## 1. 产品定位

OrbitGo 是一个 3D 太阳系旅行导航。它借用 Google Maps 的搜索、路线、图层、时间和地点详情体验，并将产品明确拆为两个入口：

- 真实航天：JPL 星历边界状态、Lambert 转移、Δv 和 C3，面向可信的任务概念比较
- 跃迁探索：矿石能源、采集、库存、载具与空间跃迁，面向游戏化旅行体验

可细化项：

- 用户角色：游客、通勤者、货运调度、科普探索
- 核心任务：去哪、何时走、怎么走、经过哪里、风险如何
- 信息密度：地图优先，面板辅助，移动端底部路线卡

## 2. 天体与地点模型

大类：

- 恒星：太阳
- 行星：水星、金星、地球、火星、木星、土星、海王星
- 卫星：月球、火卫一、木卫二、木卫三、泰坦
- 中继点：谷神星、轨道港、空间站、跃迁门
- 景点：大红斑、土星环、木卫二冰壳、泰坦甲烷湖

可细化项：

- 轨道半径、公转周期、倾角、初始相位
- 地点标签、港口名称、可登陆性、补给能力
- 星球贴图、云层、环带、冰层、城市光点

## 3. 时间与运动系统

大类：

- 全局时间：当前出发日期、时间倍率、时间滑杆
- 天体运动：行星绕太阳、卫星绕行星
- 出发窗口：根据相对位置给出游戏化评分
- 航行动画：飞船沿当前路线巡航

可细化项：

- 当前真实模式使用预生成 NASA/JPL Horizons DE441 状态向量
- 行星按 1 天、卫星按 6 小时采样，运行时线性插值
- 引入会合窗口、补给窗口、港口排班
- 添加节日、流星雨、太阳风暴等事件层

## 4. 路线规划

真实航天：

- 选择出发日期和飞行天数
- 使用出发天体在出发时刻、目标天体在到达时刻的 JPL 状态
- 太阳中心两体 Lambert 解算
- 显示出发 Δv、到达 Δv、总 Δv、C3 和候选窗口

跃迁探索：

大类：

- 推荐路线：时间、风景、风险均衡
- 最快路线：更短时间，更高风险
- 风景路线：绕行高价值景点
- 省能路线：经过中继港与补给点

可细化项：

- 引力弹弓：金星、木星等加速点
- 分段路线：轨道段、深空段、登陆段
- 多目标路线：地球到月球到火星再到木卫二
- 成本模型：矿石能源、票价、维护、舒适度

## 5. 能源与采集

- `日珀晶`：0.24 au/u，主要分布于内行星
- `赤铁核`：0.48 au/u，主要分布于岩质天体
- `冰脉晶`：0.92 au/u，主要分布于外行星冰卫星
- `雷暴氢晶`：1.80 au/u，主要分布于巨行星磁层
- `曲率晶核`：6.00 au/u，假想跃迁能源，主要分布于遥远深空区域

每个可达行星和卫星都有确定的矿脉类型、储量与单次产量。路线按 au 航程、能源单位航程和载具效率计算消耗，启航时扣除，到达后可采集目的地矿脉。

## 6. 交通工具

大类：

- 观光飞船：舒适、风景路线加成
- 太阳帆：省能、近太阳更有优势
- 急行艇：速度快、续航短
- 跃迁艇：依赖中继港，远行效率高

可细化项：

- 速度、续航、舒适度、可登陆性
- 载具外观、尾焰、航行动画
- 票价、舱位、货运能力、风险容忍度

## 7. 地图 UI 与图层

大类：

- 搜索：出发地、目的地、交换路线
- 地图总览：在地图上直接展示路线、耗时、航程、窗口评分
- 路线：模式选择、指标、分段列表
- 载具：卡片选择、速度和适配说明
- 时机：候选日期与窗口评分
- 地点：重力、大气、港口、看点
- 图层：轨道、标签、港口
- 视角：总览、路线、近景、深空第一人称
- 真实航天专属自转跟随：相机与目的天体同角速度伴飞，观察固定表面区域
- 第一人称操控台：姿态、推进器、反应堆、油门、当前能源与跃迁控制
- 启航后冻结路线几何，以任务进度驱动星历时间
- 起飞阶段使用地表第三人称镜头，离开大气后转入驾驶舱第一视角，目的地进近后转入第三人称下降镜头
- 地球和火星使用专属地表场景；未配置地表的天体保持原有轨道港出发、捕获和伴飞视角
- 高速公转卫星在航线计算后冻结出发与抵达端点，进近末段才平滑锁定目标
- 起点和终点保持画面中心，母行星与其他天体缩小并侧移为可见背景，不参与镜头追踪

可细化项：

- 标签避让、聚合、缩放层级
- 路线对比、收藏地点、历史路线
- 移动端底部抽屉、桌面端左侧路线面板

## 8. 视觉资产

大类：

- 行星/卫星材质
- 程序化拟真材质：海陆云层、云带、极冠、环形山、冰裂纹、行星环
- 真实贴图资源：优先评估 NASA 3D Resources 和 NASA Images
- 飞船、港口、中继站
- 地球海岸港与火星峡谷港两类程序化地表场景
- 航线材质、尾迹、选中高亮
- 星图背景、轨道网格、深空坐标
- 奥尔特云使用完整 360° 环状冰质天体带，不再以单点代表整个区域

可细化项：

- 当前使用 Three.js 球体、NASA 贴图、动态光照和 3D 航线
- 后续升级飞船、港口和空间站 glTF 模型
- 不同缩放级别使用不同细节
- 微信小程序中使用压缩纹理和分包加载

## 9. 数据与配置

当前原型将配置集中在 `src/app.js`：

- `BODIES`：天体与地点
- `VEHICLES`：交通工具
- `ROUTE_MODES`：路线模式
- `ENERGY_TYPES`：能源单位航程与视觉属性
- `state`：当前导航状态
- `src/astrodynamics.js`：星历读取、插值、Lambert 与轨迹采样
- `assets/ephemeris/jpl/`：预生成 JPL 星历

后续可拆分为：

```text
src/core/bodies.ts
src/core/vehicles.ts
src/core/routes.ts
src/core/time.ts
src/renderers/web-three.ts
src/renderers/wechat-three.ts
src/ui/web/
src/ui/miniprogram/
```

## 10. 开发阶段

第一阶段：

- 完成静态 Web 原型
- 验证 3D 太阳系、时间滑杆、路线与载具选择
- 确定大的分类和信息架构

第二阶段：

- 拆分核心模块为 TypeScript
- 引入构建工具、状态管理、测试
- 增强路线算法和视觉层次

第三阶段：

- 建立微信小程序工程
- 复用核心数据和路线逻辑
- 使用小程序 Canvas/WebGL 重写渲染适配层

第四阶段：

- 资产优化、性能调优、真机测试
- 接入收藏、分享、路线截图
- 按小程序包体限制做分包和懒加载
